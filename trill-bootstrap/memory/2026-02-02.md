# 2026-02-02 - Cron Model Migration & Context Overflow Issues

## Root Cause: Clawdbot Runtime System Prompt is Too Large

**Problem:** Even after optimizing workspace files (14.5KB total), Groq (128k context) still fails with "context overflow"

**Why:** Clawdbot injects a massive system prompt BEFORE workspace files:
- Full skills list (12 skills × descriptions)
- Complete tool documentation  
- Model aliases table
- Runtime configuration
- Project context injection
- Platform-specific formatting rules

**Estimated runtime injection:** 50-80k tokens minimum

## Attempted Solutions

### 1. Trimmed Workspace Files
- Split USER.md → moved interests to `kba/user-interests.md` (1.5KB)
- Moved family/food to `kba/user-family.md` (940B)
- Reduced USER.md from 6KB → 2KB
- Total workspace: only 14.5KB

**Result:** Still failing - workspace wasn't the problem

### 2. Switched Primary Model
- Gemini-2.0-flash (1M context) - hit rate limits quickly
- Back to Claude Sonnet (200k context) - works but expensive

### 3. Configured Crons for Groq
- All 10 cron jobs set to `groq/llama-3.3-70b-versatile`
- Still falling back to Opus due to context overflow

## Current State
- Primary: Claude Sonnet 4.5 (200k context)
- Crons: Attempting Groq → failing → falling back to Opus
- All cron sessions showing Opus in practice

## Options Going Forward
1. **DeepSeek R1** - 64k context, might work
2. **Minimal sub-agent config** - create stripped-down agent for crons
3. **Accept Sonnet cost** - reliable, expensive
4. **Local Ollama with preload** - keep models warm

## Key Learnings
- Workspace file size is NOT the bottleneck
- Clawdbot's core runtime prompt is the real issue
- Model fallback happens at gateway level, not controllable by cron config
- Session files cache model names but don't control actual model usage
